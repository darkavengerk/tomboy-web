# 관전 모드 스크롤백 시드 — 설계 문서

- 작성일: 2026-05-21
- 대상: 터미널 노트 **관전(spectator) 모드**
- 선행 분석: `docs/tmux-spectator-scroll-follow-feasibility.md` (구현 방식 타당성)

## 1. 배경 / 문제

관전 모드는 데스크탑의 tmux 세션을 `tmux -CC` control mode로 붙어, 활성 페인의
출력을 모바일 노트로 흘려보낸다. 그러나 **모바일에서 과거 스크롤백을 볼 수 없다.**

근본 원인: `tmux -CC` control mode는 "프로그램 출력 바이트 스트림(`%output`)"만
전달하는 프로토콜이라 페인의 *기존 스크롤백 히스토리*는 시드(seed) 시점에 한 번도
전달된 적이 없다. 시드를 만드는 `capture-pane -epJ`(브릿지
`spectatorSession.ts:319`)가 **보이는 영역(visible region)만** 캡처하기 때문이다.
그 결과 모바일 xterm의 스크롤백에는 attach 이후 누적된 `%output`만 쌓인다.

## 2. 목표 / 비목표

**목표** — 모바일 관전 화면에서 데스크탑 페인의 최근 스크롤백을 열람할 수 있게
한다. 모바일 사용자가 자기 손가락으로(네이티브 터치 스크롤) 과거를 훑는다.

**비목표**
- 데스크탑 사용자의 copy mode 스크롤 *위치* 실시간 추종 (타당성 보고서의 방식
  A) — 별건. "자리 비우고 폰으로 보기" 용도에서는 데스크탑에 스크롤할 사람이
  없어 효용이 낮다고 판단해 이번 범위에서 제외.
- 데스크탑 화면 전체 미러링 / 일반 attach 전환 (방식 C).

## 3. 채택 방식 (타당성 보고서의 "방식 B")

시드를 만들 때 `capture-pane`에 `-S` 옵션을 붙여 데스크탑 페인의 스크롤백
히스토리를 함께 캡처해 모바일 xterm 스크롤백으로 밀어 넣는다.

```
변경 전: capture-pane -epJ -t <pane>
변경 후: capture-pane -epJ -S -1000 -t <pane>
```

- `-S -1000` — 스크롤백 최근 1000줄부터 캡처. 음수 = 스크롤백, tmux가 보유분이
  더 적으면 자동 클램프(가진 만큼만 반환).
- 시드 구조는 불변: `리셋 → (alt-screen이면 토글) → 캡처 내용 → 커서 위치`.
  히스토리는 visible 영역 *앞에* 붙고, visible은 여전히 캡처 내용의 마지막
  `pane_height`줄이므로 커서 위치 계산(`CSI y;x H`)은 수정 불필요.
- 부트스트랩, 페인 전환(`switchTo`), 재연결 — 모두 `activateAndSeed` 동일 경로를
  타므로 자동으로 적용된다(각 페인의 스크롤백이 매번 다시 시드됨).

## 4. 설계 결정

- **캡처 한도 = 1000줄 (상수).** 5000줄은 단일 시드 전송량이 과다(MB대). 1000줄은
  수십~수백 KB 수준. 상수로 두고 추후 조정 가능.
- **이 한도는 데스크탑 tmux의 `history-limit`과 무관하다.** `capture-pane`은 순수
  읽기 연산이라 tmux 보관 버퍼를 변경하지 않는다. 우리가 1000줄만 읽어도 데스크탑
  사용자는 자기 tmux 설정만큼 끝까지 스크롤할 수 있다.
- **모바일 xterm `scrollback`은 기존 5000 그대로 둔다.** 요건은
  `xterm scrollback ≥ 캡처 한도`(시드가 도착 시 잘리지 않게)뿐이다. 5000을
  유지하면 1000줄 시드가 들어가고도 라이브 출력 ~4000줄의 헤드룸이 생겨, 세션
  진행 중에 시드된 과거가 곧바로 밀려나지 않는다.
- **alt-screen 페인** — alt screen에는 스크롤백이 없으므로 `-S`는 사실상 무효지만
  무해하다. 분기 없이 `-S`를 무조건 붙인다.
- **WS 프로토콜 / 클라이언트 렌더 경로 변경 없음.** 시드는 기존 `data` 프레임으로
  그대로 전달되고, 클라이언트는 받은 바이트를 `term.write`로 그릴 뿐이다.

## 5. 변경 범위

- `bridge/src/spectatorSession.ts` — `activateAndSeed`의 `capture-pane` 명령에
  `-S -<상수>` 추가. 캡처 한도 상수 1개 도입.
- 그 외 파일 변경 없음.

## 6. 위험 / 확인 사항

- 1000줄 캡처가 단일 `%begin/%end` 블록(`tmuxControlClient`)으로 버퍼링된 뒤 단일
  `data` WS 프레임으로 전달된다. 1000줄이면 허용 범위. (5000줄 이상으로 키울 경우
  청킹 검토 필요 — 이번 범위 아님.)
- `applySpectatorFit`(`transform: scale` 폭맞춤)이 긴 시드 콘텐츠에서도
  `.xterm-host`의 스크롤 경계를 정상 계산하는지 실측 확인.
- 커서 위치 — visible은 여전히 캡처 내용의 마지막 `pane_height`줄이므로
  `CSI y;x H`가 그대로 맞아야 한다. 긴 시드에서 회귀 없는지 실측 확인.

## 7. 테스트 계획

브릿지에는 단위 테스트가 없다(기존 정책). dev 브릿지 + 관전 노트로 수동 스모크:

1. 관전 노트 접속 → 모바일에서 데스크탑 페인의 과거 1000줄을 터치 스크롤로 열람 가능.
2. 데스크탑에서 페인 전환 → 새 페인의 스크롤백이 다시 시드됨.
3. 모바일 재연결(앱 토글) → 스크롤백이 다시 시드됨.
4. alt-screen 앱(claude code, vim 등) 페인 — 회귀 없음(스크롤백 없이 정상 동작).
5. 커서 위치 / `transform: scale` 폭맞춤 — 회귀 없음.

## 8. 대안 (기각)

- **방식 A — copy mode 추종.** 데스크탑의 copy mode 스크롤 위치를 `scroll_position`
  구독/폴링 + 범위 캡처로 실시간 미러. 난이도 중(4-6일). "자리 비우고 폰으로
  보기" 용도에서는 데스크탑을 스크롤할 사람이 없어 효용이 낮다고 판단 → 제외.
- **방식 C — 일반(non-control) tmux attach 전환.** 미러링 충실도는 완벽하나 관전
  모드 거의 재작성 + 단일 페인 → 윈도우 전체 렌더로 UX 후퇴 → 제외.
- **방식 D — 브릿지가 copy mode를 직접 구동.** 데스크탑 사용자의 화면을 흔든다.
  `server.ts:243-251` 주석이 이미 기각한 방향 → 제외.
