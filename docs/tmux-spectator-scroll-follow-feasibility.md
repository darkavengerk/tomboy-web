# tmux 관전 모드 — 데스크탑 스크롤 추종(follow) 구현 타당성 보고서

- 작성일: 2026-05-21
- 대상: 터미널 노트 **관전(spectator) 모드**
- 선행 문서: `docs/tmux-spectator-scroll-analysis.md` (스크롤 미갱신 원인 분석)
- 관련 파일: `bridge/src/{spectatorSession,tmuxControlClient,server}.ts`,
  `app/src/lib/editor/terminal/{wsClient,TerminalView.svelte}`

> 이 문서는 **구현 타당성 분석 전용**이다. 실제 코드 변경/플랜은 다루지 않는다.
> 목표: 데스크탑 사용자의 tmux 스크롤(copy mode)을 관전 화면이 따라가게 하려면
> 어떤 방식이 가능한지, 각 방식의 개발 난이도·완성도·위험 요소를 평가한다.

---

## 0. 전제 — 왜 지금은 안 되는가 (요약)

선행 분석의 결론을 압축하면:

- `tmux -CC` **control mode**는 "프로그램 출력 바이트 스트림(`%output`)"과 구조
  알림만 전달하는 프로토콜이다. **렌더링된 화면(스크롤 viewport 포함)을 보내지
  않는다.**
- copy mode 스크롤은 프로그램을 실행시키지 않으므로 `%output`을 만들지 않고,
  페인의 라이브 grid도 바꾸지 않는다. → 브릿지에 **0바이트가 도착**한다.
- 따라서 추종을 구현하려면 데이터 경로에 **"copy mode 상태를 능동적으로
  획득(pull / subscribe)하는 메커니즘"** 을 새로 넣어야 한다.

이 보고서는 그 "새 메커니즘"의 후보들을 평가한다.

### 활용 가능한 tmux 1차 자료(primitive)

| primitive | 용도 | 최소 tmux 버전 |
|---|---|---|
| `%pane-mode-changed %<pane>` 알림 | 페인 모드 진입/이탈 신호 | 3.0 |
| `#{pane_in_mode}` / `#{pane_mode}` | 모드 여부 / 모드 이름 | 2.x~ |
| `#{scroll_position}` | copy mode 스크롤 위치(아래에서 몇 줄 위) | 2.x~ |
| `capture-pane -p -e -J -S <s> -E <e>` | 스크롤백 포함 임의 줄 범위 캡처 | 2.x~ |
| `refresh-client -B <name>:<pane>:<format>` + `%subscription-changed` | 포맷 값 변경을 **푸시**로 구독 | 3.2 |

핵심: tmux 3.2+의 **포맷 구독(`refresh-client -B`)** 을 쓰면 `scroll_position`
변경을 폴링 없이 푸시로 받을 수 있다. 이게 가장 깔끔한 신호원이다.

## 1. 가능한 구현 방식

### 방식 A — control mode 유지 + copy mode 상태 구독/폴링 + 범위 캡처 (주력안)

현재 아키텍처(`-CC` control mode, 단일 페인 포커스 추종)를 유지한 채 **copy mode
추종 경로만 추가**한다.

동작 흐름:

1. **모드 감지.** `SpectatorSession`이 `tmuxControlClient`의 `notification`
   이벤트를 신규 구독해 `%pane-mode-changed`를 받는다. (현재
   `tmuxControlClient.ts:284`에서 미지정 태그는 `notification`으로 떨어지지만
   `SpectatorSession`이 듣지 않아 버려진다.)
   - tmux 3.2+: 더 우월한 경로 — 활성 페인에 대해
     `refresh-client -B follow:%<pane>:'#{pane_in_mode},#{scroll_position},#{alternate_on}'`
     로 **포맷 구독**을 걸고 `%subscription-changed` 푸시를 받는다. 폴링 타이머
     자체가 필요 없어진다.
   - tmux 3.0~3.1: `%pane-mode-changed`로 진입/이탈만 받고, copy mode 동안에는
     `#{scroll_position}`을 짧은 주기로 폴링.
2. **추종 모드 진입.** copy mode 감지 시 라이브 화면 렌더를 멈추고(=`%output`을
   화면에 반영하지 않음) "추종 모드"로 전환.
3. **viewport 캡처.** `scroll_position`이 바뀔 때마다, 그 위치에 해당하는
   `pane_height`줄 구간을 `capture-pane -p -e -J -S <top> -E <bottom>`으로 캡처해
   `data` 프레임(리셋 → 캡처 내용 → 커서)으로 푸시한다. `top`/`bottom`은
   `scroll_position`과 `pane_height`로 계산(음수 인덱스 = 스크롤백).
4. **추종 모드 이탈.** copy mode 종료(`%pane-mode-changed` 또는 구독 값 변화)
   시 라이브 화면을 재시드(`activateAndSeed` 재사용)하고 `%output` 스트리밍
   재개.

재사용 자산:

- `TmuxControlClient.command()` 명령/응답 머신 — 그대로.
- `activateAndSeed`의 시드 빌드 패턴(리셋 → 내용 → 커서) — 추종 프레임 빌드에 재사용.
- WS `data` 프레임 — **신규 프레임 불필요**. 클라이언트는 받은 바이트를
  `term.write`로 그릴 뿐이라 `TerminalView`/`wsClient`는 거의 손대지 않는다.
- `seeding` / `pendingOutput` 버퍼링 패턴 — 라이브/추종 전환 경합 처리에 재사용.

신규 코드: `notification`/구독 처리, copy mode 상태머신(라이브 ↔ 추종),
캡처 범위 계산, 폴백 폴링 타이머. **브릿지에 집중**(~150-300줄 추정).

> **이 방식이 사용자의 요구("따라가게")를 직접 만족시키는 현실적 경로다.**

### 방식 B — 전체 스크롤백 시드 (추종이 아닌 독립 스크롤, 저비용 대안)

추종을 구현하지 않는다. 대신 시드 시 `capture-pane`에 `-S`를 붙여 페인의 과거
히스토리를 **한 번에** 클라이언트 xterm 스크롤백으로 밀어 넣는다.
(`spectatorSession.ts:319`의 `capture-pane -epJ` → `capture-pane -epJ -S -2000`
수준 변경. 클라이언트 xterm은 이미 `scrollback: 5000`.)

그러면 모바일 사용자가 *자기 손가락으로* 진짜 히스토리를 스크롤할 수 있다.
데스크탑이 어디를 보든 무관하게, 모바일은 같은 과거 데이터에 접근 가능.

- 사용자가 명시한 "데스크탑을 따라가기"는 아니다.
- 그러나 그 이면의 실제 욕구("모바일에서 스크롤백을 보고 싶다")는 상당 부분
  해소한다. 그리고 방식 A와 **공존 가능**(A 도입 전 단독으로, 또는 A의
  히스토리 보강용으로).

### 방식 C — 일반(non-control) tmux attach로 전환 (이론상 완벽, 큰 피벗)

`-CC` control mode를 버리고 일반 `tmux attach`로 붙는다. 일반 클라이언트는 tmux가
**렌더링한 화면**을 escape sequence로 받는다 → copy mode가 그대로 그려진다.
데스크탑이 스크롤하면 그 화면이 우리에게도 그려진다. **추종이 "공짜"로 된다.**

기존 사이즈 트릭(`stty cols 500 rows 200` + 타깃 `window-size smallest`)은 일반
attach에도 그대로 적용 가능.

대가:

- 일반 클라이언트는 **페인이 아니라 윈도우 전체**(상태줄 + 모든 페인 + 분할선)를
  그린다. 현재 관전 모드의 "포커스된 단일 페인 풀스크린" UX가 사라진다 —
  모바일에서 셀이 작아진다.
- 현재 `%window-pane-changed` 기반 포커스 추종, `pane-switch`/`pane-resize`
  프레임, `send-keys -H` 입력, `« ‹ › »` 네비게이션이 전부 control mode 명령에
  묶여 있다 → **전부 재배선**. (네비/입력은 raw pty 쓰기로 오히려 단순해질 수도
  있으나 어쨌든 재작성.)
- `tmuxControlClient.ts` + `spectatorSession.ts`의 상당 부분이 폐기/대체.

사실상 관전 모드 재작성이다.

### (기각) 방식 D — 브릿지가 copy mode를 직접 구동

브릿지가 `send-keys -X scroll-up` 등으로 copy mode를 *주도*. copy mode의 스크롤
위치는 페인 단위로 공유되므로 **브릿지가 스크롤하면 데스크탑 사용자의 화면도 같이
움직인다.** `server.ts:243-251` 주석이 이미 이 이유로 명시적으로 기각한 방향이며,
"추종"이 아니라 그 반대(주도)다. 검토 대상에서 제외한다.

## 2. 개발 난이도

| 방식 | 난이도 | 추정 공수 | 변경 범위 |
|---|---|---|---|
| A (구독/폴링 + 범위 캡처) | **중** | 4-6일 | 브릿지 중심, 클라이언트 거의 무변경 |
| B (전체 스크롤백 시드) | **하** | 0.5-1일 | 브릿지 1~수 줄 + 클라이언트 동작 확인 |
| C (일반 attach 전환) | **상** | 1-2주+ | 관전 모드 거의 재작성, 클라이언트 UX 변경 |

**방식 A 난이도 내역:**

- 쉬운 편: `TmuxControlClient`에 명령/응답·이벤트 인프라가 이미 있다. `data`
  프레임을 재사용하므로 WS 프로토콜·클라이언트는 손대지 않아도 된다.
- 까다로운 편:
  - 라이브 ↔ 추종 **상태머신**과 그 사이 `%output` 경합 처리(이탈 시 재시드
    순서 보장).
  - `scroll_position` → `capture-pane` 줄 범위 변환의 **off-by-one / 음수
    인덱스 의미**를 타깃 tmux 버전에서 실측 검증해야 함.
  - tmux 3.2+ 구독 경로와 3.0~3.1 폴링 폴백, 2.x 미지원 — **2~3개 코드
    경로**를 분기·테스트.
- 브릿지에는 단위 테스트가 없다(터미널 노트 skill 명시). control 프로토콜
  파서(`tmuxControlClient`)는 순수 함수라 테스트 가능하지만, 상태머신 검증은
  실제 tmux 대상 수동 스모크 테스트에 의존하게 된다 → 검증 비용이 공수에 포함.

**방식 B 난이도 내역:** `capture-pane`에 `-S` 추가가 본질의 전부. 시드가 길어질
때 클라이언트 xterm이 정상 동작하는지(스크롤백 한도, `applySpectatorFit`의
스크롤 경계 계산) 확인만 하면 된다.

**방식 C 난이도 내역:** 신규 코드보다 **회귀 위험 관리**가 비용의 핵심.
입력·네비·종료·사이즈 트릭·포커스 유지 등 현재 동작하는 모든 것을 새 기반 위에서
다시 맞춰야 한다.

## 3. 완성도 전망

"완성도"를 두 축으로 본다 — **(가) 데스크탑 추종 충실도**, **(나) 현재 제품
정의(단일 페인 풀스크린 관전) 유지 여부**.

| 방식 | (가) 추종 충실도 | (나) 현 제품 정의 유지 | 종합 체감 |
|---|---|---|---|
| A | ~75-85% | 유지 | 실용적으로 충분 |
| B | 0% (추종 아님) | 유지 | "스크롤백 열람" 욕구는 충족 |
| C | ~100% | **회귀**(윈도우 전체 렌더) | 미러링은 완벽, UX는 후퇴 |

**방식 A 완성도 상세:**

- 스크롤 *위치*를 따라간다. 지연: 구독 경로면 tmux의 구독 재평가 주기(보통
  1초 미만), 폴링 폴백이면 폴링 주기(<200ms 권장).
- 매 변경마다 viewport 전체를 재캡처·재렌더하므로 **부드러운 스크롤이 아니라
  "탁탁" 스냅 갱신**. 데스크탑 사용자가 천천히 읽으며 스크롤하는 시나리오엔
  충분하지만, 빠르게 휙휙 넘기면 중간 프레임이 생략되고 점프하듯 보인다.
- **미반영 요소**: copy mode의 일시적 **선택 영역 하이라이트**, copy mode
  **검색 하이라이트**, copy mode 커서. `capture-pane -e`는 SGR 색은 주지만 이
  일시 상태들은 못 준다.
- 결론: "데스크탑이 스크롤한 데를 나도 본다"는 ~75-85% 만족. "데스크탑 화면을
  픽셀 단위로 미러링"은 아님.

**방식 B 완성도 상세:** 추종은 0%. 그러나 모바일에서 실제 페인 히스토리를
독립적으로 열람하는 기능으로서는 거의 완성형. 한계는 시드 시점 스냅샷이라
이후 데스크탑이 더 쌓은 히스토리는 다음 페인 전환/재연결 때까지 반영 안 됨.

**방식 C 완성도 상세:** tmux가 직접 그리므로 copy mode·선택·검색·부드러운
스크롤이 전부 그대로 온다 — 미러링 충실도는 사실상 100%. 단 단일 페인 풀스크린이
아니라 윈도우 전체가 와서, 모바일 가독성·현재 UX 기준으로는 후퇴다.

## 4. 잠재적 위험 요소

### 4.1 공통 위험

- **tmux 버전 의존성.** `%pane-mode-changed`(3.0+), 포맷 구독 `refresh-client
  -B`/`%subscription-changed`(3.2+), `capture-pane -S` 음수 인덱스 의미가
  버전마다 다를 수 있다. 관전 모드는 이미 `send-keys -H`로 tmux 3.0+를 요구하지만
  copy mode 추종은 3.2+에서야 최적 경로가 열린다. 구버전 폴백 정책(폴링 / 또는
  "미지원" 명시)을 정해야 한다.
- **copy-mode-vi vs emacs / 사용자 커스텀 키테이블.** viewport는
  `scroll_position`으로 결정되므로 키테이블과 무관해야 하지만, 실측 검증 필요.
- **alt-screen 페인의 copy mode.** claude code·vim 같은 alt-screen 앱 위에서
  copy mode는 스크롤백이 거의 없다. 추종해도 보여줄 게 없는 엣지 케이스 —
  `alternate_on`을 함께 봐서 분기하는 게 안전.
- **중첩 tmux.** 안쪽 tmux의 copy mode는 바깥 페인 입장에선 그냥 바이트
  스트림이라 추종 불가. 바깥 tmux의 copy mode만 추종 가능 — 기존 관전 모드
  불변식과 동일한 한계.

### 4.2 방식 A 고유 위험

- **부하.** copy mode 동안만 활성화되므로 idle 비용은 0이다. 그래도 동시
  관전자 수 × 갱신 빈도 × `capture-pane`(viewport 텍스트량)가 control 채널과
  tmux 서버에 부하. **완화책**: `scroll_position`은 싸게 구독/폴링하고, 값이
  실제로 바뀐 경우에만 비싼 `capture-pane`를 발행한다.
- **라이브 vs 추종 경합.** copy mode 중에도 페인 프로그램이 출력하면 `%output`은
  계속 도착한다. 추종 화면에 끼어들지 않게 억제하고, copy mode 이탈 시 라이브
  재시드 순서를 보장해야 한다(기존 `seeding`/`pendingOutput` 패턴 재사용).
- **이중 스크롤 혼란.** 추종 중에는 모바일 네이티브 xterm 스크롤이 의미를 잃는다
  (브릿지가 화면을 통째로 갈아끼우므로). 추종 중 모바일 스크롤 UX를 어떻게 할지
  (잠금? 안내?) 정의가 필요.
- **상태 누수.** WS 종료/페인 전환 시 구독(`refresh-client -B`) 해제와 폴링 타이머
  정리를 빠뜨리면 tmux 측 구독이 남거나 좀비 타이머가 생긴다.
- **off-by-one.** `scroll_position` → 캡처 줄 범위 변환 오차는 화면이 한 줄
  어긋나 보이는 미세 버그로 직결. 버전별 실측 검증이 위험 완화의 핵심.

### 4.3 방식 B 고유 위험

- 낮음. 시드가 길어지면 초기 전송량·xterm 메모리가 늘고, `applySpectatorFit`의
  스크롤 경계(`.xterm-host` overflow) 계산이 큰 콘텐츠에서 정상인지 확인 필요.

### 4.4 방식 C 고유 위험

- **대규모 재작성 회귀.** 입력(`send-keys -H`)·네비(`« ‹ › »`)·종료 처리·사이즈
  트릭·포커스 유지가 전부 새 기반으로 옮겨가며, 각각이 회귀 후보.
- **UX 후퇴.** 단일 페인 풀스크린 → 윈도우 전체. 모바일에서 가독성 저하.
- **상태줄/프리픽스 키 등 부수 렌더**가 의도치 않게 따라온다.

## 5. 비교 요약 및 권고

| 항목 | 방식 A | 방식 B | 방식 C |
|---|---|---|---|
| 사용자 요구("따라가기") 충족 | O (~75-85%) | X (독립 스크롤) | O (~100%) |
| 개발 난이도 | 중 (4-6일) | 하 (~1일) | 상 (1-2주+) |
| 변경 범위 | 브릿지 중심 | 브릿지 미소 | 관전 모드 재작성 |
| 현 제품 정의 유지 | 유지 | 유지 | 회귀 |
| 핵심 위험 | 버전 분기·경합·off-by-one | 거의 없음 | 대규모 회귀·UX 후퇴 |

**권고(분석 결론, 설계는 별도):**

1. 사용자가 명시한 "스크롤을 따라가게"의 현실적 답은 **방식 A**다. 기존
   아키텍처를 보존한 채 브릿지에 copy mode 추종 경로만 더하고, `data` 프레임을
   재사용하므로 클라이언트 변경이 거의 없다. 완성도 ~75-85%는 "데스크탑이
   스크롤한 데를 모바일에서 본다"는 목적에 실용적으로 충분하다.
2. **방식 B는 방식 A와 별개로, 또는 그 전에 단독 도입할 가치가 있다.** 공수가
   하루 수준이고 위험이 거의 없으며, "모바일에서 스크롤백을 본다"는 이면의 욕구를
   즉시 해소한다. A의 선행 디리스킹 단계로도 적합.
3. **방식 C는 충실도는 완벽하지만 비용·위험이 과도하다.** 관전 모드를
   "단일 페인 추종"에서 "윈도우 전체 미러링"으로 제품 정의 자체를 바꾸려는
   결정이 선행될 때만 검토할 가치가 있다.

가장 큰 단일 위험은 **tmux 버전 분기**다 — 추종의 최적 경로(포맷 구독)는
3.2+에서만 열리므로, 구버전 폴백 정책을 설계 단계에서 먼저 못박아야 한다.
