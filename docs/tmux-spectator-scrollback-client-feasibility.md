# 관전 모드 스크롤백 — 클라이언트 수정 타당성 보고서

- 작성일: 2026-05-21
- 대상: 터미널 노트 **관전(spectator) 모드** — 모바일에서 스크롤백 열람
- 선행: `docs/tmux-spectator-scrollback-seed-design.md`(방식 B 설계),
  `docs/tmux-spectator-scroll-analysis.md`(원인 분석)
- 요구: 스크롤백 열람이 **멈춘 패널 / 스트리밍 중 둘 다 동등하게** 동작할 것

> 이 문서는 **타당성 분석 전용**이다. "실제 제대로 동작하지 않을 가능성"을
> 명시적으로 포함해 평가한다.

---

## 0. 배경 — 지금 왜 안 되는가 (요약)

브릿지 변경(`capture-pane -S -1000`)으로 스크롤백 1000줄이 시드에 담겨
`term.write()`로 **xterm 내부 버퍼에는 정상 도착**한다. 그러나 관전 모드의
스크롤 면(`.xterm-host` overflow-y)은 `applySpectatorFit`이 `.xterm-screen`
높이(= 보이는 화면 `rows × cellHeight`)로만 크기를 잡는다. xterm 스크롤백은
`.xterm-screen`를 늘리지 않으므로 그 1000줄은 버퍼에 **갇혀** 있다. 따라서
수정은 **클라이언트 쪽**이어야 한다.

---

## 1. 먼저 — 무엇을 "스크롤백"이라 부르는가 (적용 범위의 근본 한계)

이게 타당성의 절반을 가른다. tmux 페인은 두 종류다.

### 1a. 일반 화면(normal screen) 패널 — 스크롤백 있음

셸, 로그 tail, 빌드 출력, 그리고 **claude code** 등. 출력이 새 줄을 찍으면
위로 밀려 xterm 스크롤백 버퍼에 쌓인다. **이 패널이 이 기능의 진짜 대상이다.**

claude code가 여기 속한다는 점이 중요하다 — claude code는 alt screen을 쓰지
않는다. 끝난 메시지는 일반 화면에 *확정 출력*으로 남아 터미널 스크롤백에
쌓이고, 맨 아래 라이브 영역(입력창·스트리밍 중인 응답)만 제자리에서 다시
그린다. 그래서 데스크탑 터미널을 스크롤하면 지난 대화가 보인다. 관전 모드의
xterm도 시드 + `%output`으로 claude code의 같은 히스토리를 버퍼에 그대로
쌓는다 — 스크롤 UI가 도달을 못 할 뿐, 히스토리 자체는 거기 있다.

### 1b. 대체 화면(alt screen) 앱 — 스크롤백 **없음**

vim, htop, less, nano, tmux copy mode 등 풀스크린 TUI. alt screen은 보이는
영역을 제자리에서 다시 그릴 뿐 스크롤백이 **원천적으로 존재하지 않는다**.
`capture-pane -S`도 alt 화면만 반환한다. 어떤 클라이언트 설계로도 없는
히스토리를 만들어낼 수 없다.

**그러나** — alt-screen 앱에서 "위로 보기"는 *다른 경로로 동작한다*. 앱
자체가 스크롤 가능한 UI(less 페이지 등)를 갖고, 보내기 팝업의 `PgUp`/`PgDn`
(또는 데스크탑 키보드)으로 스크롤 키를 보내면 → 앱이 alt 화면을 다시 그림 →
그 redraw가 `%output`으로 관전 화면에 전달된다.

**판별 기준:** 일반 vs 대체 화면은 `\e[?1049h`/`\e[?1049l`로 전환되며,
브릿지가 이미 `#{alternate_on}`으로 추적한다(시드의 `altScreen` 플래그). 즉
런타임에 어느 쪽인지 이미 알 수 있다.

**결론:** 이 기능의 실제 범위는 **"일반 화면 패널의 스크롤백 열람"** 이며,
**claude code를 포함한다** — 관전 모드의 대표 용도가 곧 이 기능의 대상이다.
범위 밖은 풀스크린 TUI(vim·htop·less)뿐이고, 그쪽 "스크롤"은 send-keys 경로로
별도 동작한다.

## 2. "둘 다 동등하게"의 근본 긴장과 해법

관전 화면은 **살아있는 미러**다. "과거를 읽기"와 "현재를 미러링"은 본질적으로
충돌한다 — 출력이 흐르는 동안 위로 스크롤해 고정해두려 하면, 버퍼가 발밑에서
계속 자란다.

다행히 이 충돌에는 **업계 표준 해법**이 있다 (Slack, 터미널 에뮬레이터, 로그
뷰어 전부 같은 방식):

> **위로 스크롤하면 라이브 추종을 자동 해제(freeze)** 한다. 그 동안 새 출력은
> 버퍼에 계속 쌓이되 화면은 사용자가 둔 위치에 고정. "↓ 새 출력 N줄 · 맨
> 아래로" 인디케이터를 띄우고, 탭하면 맨 아래로 점프 + 추종 재개.

xterm.js는 이 동작의 **절반을 기본 제공**한다 — 사용자가 스크롤백으로 올라가
있으면 `term.write()`가 들어와도 강제로 맨 아래로 끌어내리지 않고 위치를
유지한다. 우리가 추가할 것은 (1) 스크롤 면을 스크롤백까지 노출, (2) "새 출력
N줄 / 맨 아래로" 인디케이터 UI.

이 패턴이면 **"멈춘 패널"과 "스트리밍 중" 둘 다** 깔끔하게 처리된다 — 멈춘
패널은 자유 스크롤, 스트리밍 중엔 위로 올리는 순간 freeze되어 방해 없이 읽고,
인디케이터로 라이브 복귀. "둘 다 동등하게"의 현실적 답이다.

주의: 이건 *완벽한 동시성*("위를 읽으면서 아래 라이브도 실시간으로")이
아니다. 그건 살아있는 터미널 미러에서는 달성 불가능하다. freeze 패턴은
"한 번에 한 모드"로 둘 다를 *깔끔하게* 만든다. 동시 표시를 고집하면 타당성은
"불가" 쪽이다.

## 3. 구현 접근법

두 접근 모두 위 §2의 freeze + 인디케이터 UX 레이어를 그 위에 얹는다.

### 접근 A — xterm 네이티브 뷰포트 스크롤 활용

xterm 자체의 `.xterm-viewport`(내부 overflow-y 스크롤 요소)가 스크롤백 스크롤을
담당하게 한다. `term.scrollLines/scrollToLine/onScroll` API(설치된 xterm v6에
존재 확인)를 쓴다. 관전 모드의 width-fit `transform: scale`이 xterm 뷰포트
스크롤과 부딪히지 않도록 3-레이어 DOM을 재조정.

- 장점: xterm 내장 동작(스크롤백 유지, 스크롤백 한도 관리)을 그대로 활용 —
  새로 만들 상태가 적음.
- 위험: `transform: scale`된 조상 안의 스크롤 요소를 **모바일에서 터치로
  스크롤**할 때의 동작이 미지수. 기존 코드가 `zoom`을 버리고 `transform:
  scale`로 간 게 정확히 이런 모바일 렌더 버그 때문 — 스크롤 면을 추가하는 건
  새 미지수다. **§5 스파이크 없이는 확정 불가.**

### 접근 B — 커스텀 스크롤 면 + `scrollToLine` 동기화

`.xterm-host`를 단일 스크롤 면으로 유지하되, `.xterm-stage` 높이를 보이는
화면이 아니라 **전체 버퍼**(`term.buffer.active.length × cellH × scale`)로
잡고, `.xterm-host`의 scrollTop → 버퍼 줄 변환 → `term.scrollToLine()` 호출.
마운트는 `position: sticky`로 뷰포트에 고정.

- 장점: 스크롤 면이 하나(통일된 UX), width-fit과 한 좌표계.
- 위험: 라이브 출력마다 버퍼 길이가 변함 → stage 높이 재계산 → 사용자가
  스크롤백에 올라가 있으면 **스크롤 위치가 픽셀 기준으로 어긋남**. 줄(line)
  기준 앵커링으로 보정해야 하며 구현이 까다롭다. sticky 마운트 + 모바일 관성
  스크롤 + 프로그래매틱 `scrollToLine` 재동기화가 서로 미끄러질 수 있음.

## 4. "제대로 동작하지 않을 가능성" — 솔직한 위험 목록

사용자가 명시적으로 요청한 항목. 코드 분석만으로는 확정 불가한 것들을 표시한다.

| # | 위험 | 심각도 | 코드만으로 확정 가능? |
|---|------|--------|----------------------|
| R1 | `transform: scale` 조상 안의 스크롤 요소를 모바일 Safari/Chrome에서 터치 스크롤할 때 깨질 수 있음 (셀 위치 어긋남·스크롤 먹통). 기존 `zoom→scale` 전환 흉터가 있는 영역. | **높음** | ❌ 실기기 스파이크 필요 |
| R2 | 모바일 관성(momentum) 스크롤 + 프로그래매틱 재동기화(`scrollToLine`)가 서로 튕김 — 스크롤이 끊기거나 튀는 느낌. | 중 | ❌ 실기기 |
| R3 | xterm **DOM 렌더러**가 스크롤 프레임마다 행을 다시 그림 — 중저가 폰에서 스크롤 버벅임 가능. | 중 | ❌ 실기기 |
| R4 | (접근 B) 라이브 출력으로 버퍼가 자라며 stage 높이가 변해 스크롤 위치가 어긋남 — 줄 앵커링 필요. | 중 | ⚠️ 설계로 완화 |
| R5 | 프로그램이 `clear`/`\e[3J`/RIS를 내면 xterm 스크롤백이 **세션 중에 날아감**. tmux copy mode도 같은 한계 — 불가피. | 낮음(불가피) | ✅ |
| R6 | **페인 전환 시 재시드(RIS)** 가 스크롤백·스크롤 위치를 리셋. 새 패널이니 일면 타당하나, 포커스가 바뀔 때마다 열람 상태가 사라짐. | 낮음(설계상) | ✅ |
| R7 | 시드는 attach 시점의 **과거 1000줄만** 운반. 그 이전 히스토리는 도달 불가. | 낮음(설계상) | ✅ |
| R8 | §1b — alt-screen 앱엔 스크롤백이 없어 기능이 **적용 자체가 안 됨**. (단 send-keys 경로로 별도 동작.) | — (범위 밖) | ✅ |

R1~R3이 핵심이다. **이 셋은 실기기에서 직접 만져보기 전엔 "될지 안 될지"를
단언할 수 없다.** 이것이 사용자 질문 — "실제 제대로 동작하지 않을 가능성" —
에 대한 가장 정직한 답이다: *가능성은 있고, 그 가능성은 디바이스 스파이크로만
해소된다.*

## 5. 권고 — 설계 전에 디바이스 스파이크부터

R1~R3 때문에, 전체 설계를 확정하기 전에 **작은 스파이크(prototype)** 를 먼저
권고한다:

> 관전 모드의 `transform: scale`된 3-레이어 DOM 안에서, 실제 모바일 기기로
> xterm 스크롤백을 터치 스크롤할 수 있는지 — 접근 A(네이티브 뷰포트)를 최소
> 코드로 붙여 검증. 1000줄 더미 시드를 넣고 폰에서 끌어본다.

- 스파이크가 **성공** → 접근 A로 본설계 진행 (freeze/인디케이터 UX 얹기).
- 스파이크가 **실패**(터치 스크롤이 깨짐) → 접근 B로 전환하거나, 접근 B도
  R2/R3가 심하면 "관전 모드에 별도 *히스토리 보기* 모드"(라이브를 멈추고
  스냅샷을 일반 스크롤로 — copy mode를 앱 모드로 명시화) 같은 더 보수적인
  대안을 검토.

스파이크는 본설계의 1번 컴포넌트로 넣는다. 결과에 따라 나머지 설계가
가지치기되도록.

## 6. 타당성 종합 판정

| 항목 | 판정 |
|------|------|
| 일반 화면 패널, 멈춤 | **타당** — 확실히 가능 |
| 일반 화면 패널, 스트리밍 중 | **조건부 타당** — freeze + 인디케이터 패턴으로 *깔끔하게* 가능. "동시 표시"는 불가. |
| claude code (일반 화면 앱) | **타당 — 범위 안.** 관전 xterm이 시드+`%output`으로 히스토리를 쌓음. 스크롤 UI만 고치면 됨. |
| 풀스크린 TUI(vim·htop·less) | **범위 밖** — alt screen은 스크롤백이 없음. send-keys 경로로 별도 동작(기존). |
| "둘 다 동등하게" 요구 충족 | **충족 가능** — 단 "한 번에 한 모드"(freeze 패턴)로. 완전 동시성은 아님. |
| 핵심 불확실성 | R1~R3 — `transform: scale` + 모바일 터치 스크롤. **디바이스 스파이크 전엔 단언 불가.** |

**한 줄 결론:** 클라이언트 수정은 **타당하다 — 단 두 가지 정직한 단서와
함께**. (1) "둘 다 동등하게"는 *동시 표시*가 아니라 *freeze 패턴*("위로
스크롤 = 라이브 일시정지 + 새 출력 N줄 인디케이터")으로 달성된다. (2) 모바일
`transform: scale` 안의 터치 스크롤이 실제로 매끄러운지는 코드로 단언할 수
없어, 본설계 1단계로 **디바이스 스파이크**를 두어 R1~R3를 먼저 해소해야 한다.
스파이크가 깨지면 더 보수적인 "히스토리 보기 모드"로 후퇴하는 길도 열어둔다.
